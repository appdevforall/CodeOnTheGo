# ADFA-4128 spike retro — why the human time was disproportionate

Four agents audited the full pre-compaction transcript (07-08 → 07-14, ~13.6k events). Bryan's question: why did getting this presentable take so much of *my* (Bryan's) hands-on time, where did the agent cut corners / not actually test, and how much of that judgment can the agent own going forward?

## The shape of the time

Measured keyboard hands-on per day (a floor — it excludes off-keyboard reload-watching and any editing done outside chat): 07-08 0.9h · 07-09 0.4h · 07-10 1.2h · 07-11 0.3h · 07-13 0.6h · 07-14 **2.5h**. Two corrections to my first draft: (1) 07-14's heavy load was the on-device wiring/debug/git cleanup, not doc-ification — I'd wrongly used the raw message count to imply the latter; (2) the doc write-up's real cost was ~4h, not the 2.1h keyboard floor, because most of Bryan's doc editing happened directly in Confluence, off-transcript, where the floor can't see it. So the two biggest costs were the on-device wiring (07-14, ~2.5h, all in chat) and the doc polish (~4h, mostly off-transcript) — the doc really did rival the hands-on of the rest of the spike, matching Bryan's felt sense. That doc-polish friction is what this retro is about.

## One root cause under almost every corner-cut

**A tool/log/build success was treated as "the target behavior is verified."** The gap only showed when someone looked at the actual screen, ran the specific case that mattered, or asked a pointed question. Examples:

- "Tier-1 hot-swap succeeded" — claimed from a service-log line; next screenshot showed the OLD text. Twice.
- "incremental compile integrated & working" — validated against a no-op edit (searched for strings not in the file); clean green run, nothing changed.
- "on-device loop works, no Mac in the loop" — the demo was actually Mac-triggered; admitted unprompted 9h later.
- Stuck "Compiling…" banner chased ~15 min through daemon/shell theories — the answer was one logcat line: `Color.parseColor("#A00")` (invalid hex) crashed render.
- "total reload ≈1s at 30k LoC" — only the compile stage was measured; dex was never benchmarked.
- Team doc: "game state survives every reload" — that's just SharedPreferences, not the mechanism. (Bryan caught it.)

The two Bryan personally caught were both already packaged into a **team-facing doc** — the highest-cost place for this pattern to surface.

## Why the doc took so long

- **Wrong formalism for the audience** — defaulted to ADR/decision-register structure (Context/Alternatives/Decision, B1–B6, Option 0/1/2/A/B/C) for a team-share narrative; Bryan collapsed it into one chronological table.
- **Verbosity not fixed on first correction** — mid-sentence bold + run-on prose needed killing across two separate rounds; recurred in the child-ticket drafts.
- **Tooling churn became manual labor** — mermaid doesn't render in Confluence + no attachment API → Bryan hand-dragged every diagram and the video; live-doc flush races clobbered edits twice; whole-body-replace blocked me doing section edits myself.
- **Doc-code drift** — README sat on "phase 1" while the spike was at phase 7.

## Also recurring, self-inflicted

- **On-device/Mac architecture invariant violated 5×** across the week — stated clearly once, then silently reintroduced (Mac-side poller, Mac-triggered demo) until "Stop Stop Stop" and a model switch. Invariants stated once get lost across compaction/model-switches unless written into DESIGN/README immediately and re-checked each phase.
- **`git add -A` habit** — committed ~1.4GB of build assets + demo mp4s + settings.local.json into history (would've been rejected by GitHub's 100MB limit); required filter-branch surgery. The repo convention already warns against blanket-staging for exactly this reason; it recurred anyway.

## How much of this judgment the agent can own

Most of what cost Bryan time was **not taste** — it was standing rules already written in this repo's CLAUDE.md / skills that the agent failed to self-enforce. Those are now durable memory:

- `verify-behavior-not-green-build` — observe the real output, run the case that matters, debug cheapest-signal-first, state what wasn't verified. Proactively.
- `terse-team-docs-by-default` — one structure + concise prose from the start; no spec formalism; low bold density.
- `ondevice-loop-runs-without-mac` (pre-existing) — plus the general habit: re-assert stated invariants after every compaction / model switch.
- Self-trigger the already-mandated gates without being asked: code-review pass, docs/README freshness, android-qa for UI, full-diff-before-push, `git status` before each commit.

**Genuinely NOT delegable** (rare, and correctly Bryan's): the ambitious target + product vision, what to demo and how to narrate it, doc structure for how *this team* discusses tradeoffs, deadlines, device/credential/ticket access, and the appdevforall push greenlight.

The honest split: the repeated, time-eating corrections were self-enforceable rules the agent should own; the taste/access calls that genuinely needed Bryan were comparatively few.
