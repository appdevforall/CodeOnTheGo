XKCD Plugin — Code Walkthrough

A Tier 3 documentation page for the random-xkcd-plugin module — the canonical "this is what a small Code on the Go plugin looks like" example.

Goal of this plugin: demonstrate the bottom-sheet plugin surface, network IO with offline fallback, image rendering, the clipboard, and the IDE's three-tier tooltip / docs system. Under 300 LOC of Kotlin + a handful of XML resources.

1. Plugin entry point

Every plugin implements IPlugin for lifecycle, then opts into one or more extension interfaces. XkcdRandomPlugin implements two of them: UIExtension for the bottom-sheet tab, and DocumentationExtension for this very page.

class XkcdRandomPlugin : IPlugin, UIExtension, DocumentationExtension {

    override fun getEditorTabs(): List<TabItem> = listOf(
        TabItem(
            id = "xkcd_bottom_tab",
            title = "XKCD",
            fragmentFactory = { XkcdPanelFragment() },
            order = 200
        )
    )

    override fun getTier3DocsAssetPath(): String? = "docs"
}

The IDE walks src/main/assets/docs/ at install time and serves each file at http://localhost:6174/plugin/<pluginId>/<path>. Reference files with relative paths (e.g. css/walkthrough.css).

2. Manifest metadata + permissions

<meta-data
    android:name="plugin.permissions"
    android:value="network.access,filesystem.read" />

Permissions live in AndroidManifest.xml as comma-separated values. network.access is enforced by PluginSecurityManager for any java.net.URL use; filesystem.read is required to read the disk cache on cold start. Request only what you need.

3. Network: XkcdApiClient.kt

Two endpoints, no auth, no rate-limit headers. We use OkHttp because it's small and the API is famously readable.

class XkcdApiClient(
    private val client: OkHttpClient = defaultClient(),
) {
    fun fetchRandom(): XkcdComic? {
        val latest = fetchLatest() ?: return null
        repeat(2) {
            val pick = Random.nextInt(1, latest.num + 1)
            if (pick == 404) return@repeat   // xkcd #404 returns HTTP 404 on purpose
            fetchByNumber(pick)?.let { return it }
        }
        return latest
    }

    fun fetchLatest(): XkcdComic? = getJson("https://xkcd.com/info.0.json")?.let(::parseComic)
    fun fetchByNumber(num: Int): XkcdComic? =
        getJson("https://xkcd.com/$num/info.0.json")?.let(::parseComic)
}

The fetcher uses Android's bundled org.json.JSONObject rather than pulling in Moshi/kotlinx-serialization — small dependency footprint keeps the demo readable.

4. Cache: XkcdDiskCache.kt

One PNG + one JSON file under cacheDir/xkcd/. Bounded at 5 MB (review-checklist theme #1: resource bounds on asset reads).

class XkcdDiskCache(context: Context) {
    private val dir = File(context.cacheDir, "xkcd").apply { mkdirs() }
    private val metaFile = File(dir, "last.json")
    private val imageFile = File(dir, "last.png")

    fun save(comic: XkcdComic, pngBytes: ByteArray) {
        if (pngBytes.size > MAX_IMAGE_BYTES) return  // bounded
        metaFile.writeText(toJson(comic))
        imageFile.writeBytes(pngBytes)
    }

    companion object { const val MAX_IMAGE_BYTES = 5 * 1024 * 1024 }
}

5. UI: XkcdPanelFragment.kt

Standard Fragment, with one plugin-specific quirk: the layout inflater must be wrapped via PluginFragmentHelper, otherwise plugin-side R.layout.* won't resolve.

override fun onGetLayoutInflater(savedInstanceState: Bundle?): LayoutInflater {
    val inflater = super.onGetLayoutInflater(savedInstanceState)
    return PluginFragmentHelper.getPluginInflater(
        XkcdRandomPlugin.PLUGIN_ID,
        inflater
    )
}

The fetch path is a coroutine off viewLifecycleOwner.lifecycleScope: the scope auto-cancels in onDestroyView(), so we don't have to track jobs by hand.

private fun loadRandomComic() {
    if (loadJob?.isActive == true) return  // skip rapid-tap fan-out
    if (currentComic == null) showLoading()  // avoid cached-comic flash
    loadJob = viewLifecycleOwner.lifecycleScope.launch {
        val (comic, bmp) = withContext(Dispatchers.IO) {
            fetchDecodeAndCache()  // network + decode both on IO
        } ?: run { showFailure(); return@launch }
        showComic(comic, bmp)
    }
}

6. Gestures: TapCountClassifier.kt

Why not android.view.GestureDetector? It resolves single and double tap, but not triple. The ticket calls for triple. A 25-line state machine is also easier to teach and unit-test than nesting behavior on top of GestureDetector.

class TapCountClassifier(
    private val windowMillis: Long = DEFAULT_WINDOW_MS,  // 300 ms
) {
    enum class Classification { SINGLE, DOUBLE, TRIPLE }

    fun onTap(now: Long): Boolean {
        if (count == 0 || now - lastTapAt > windowMillis) count = 1
        else                                              count++
        lastTapAt = now
        return count >= 3   // 3+ closes the burst eagerly
    }

    fun resolve(): Classification? = when {
        count <= 0 -> null
        count == 1 -> Classification.SINGLE
        count == 2 -> Classification.DOUBLE
        else        -> Classification.TRIPLE   // 4+ clamps to TRIPLE
    }
}

Pure Kotlin, no Android imports — so the unit tests run in plain JUnit (no Robolectric):

@Test
fun `four taps clamp to TRIPLE`() {
    val c = TapCountClassifier(windowMillis = 300L)
    c.onTap(now = 1000L); c.onTap(now = 1100L)
    c.onTap(now = 1200L); c.onTap(now = 1300L)
    assertEquals(Classification.TRIPLE, c.resolve())
}

Scroll vs. tap. The fragment's OnTouchListener sits on a ScrollView, so every fling/scroll also ends in an ACTION_UP. To keep scroll-end from being classified as a tap, track the ACTION_DOWN coordinates and only feed the classifier when the finger moved less than the system touch slop:

val slop = ViewConfiguration.get(view.context).scaledTouchSlop
root.setOnTouchListener { _, e ->
    when (e.actionMasked) {
        MotionEvent.ACTION_DOWN -> { downX = e.x; downY = e.y }
        MotionEvent.ACTION_UP -> {
            val dx = e.x - downX; val dy = e.y - downY
            if (dx*dx + dy*dy <= slop*slop) handleTap()
        }
    }
    false  // never consume — let the ScrollView keep scrolling
}

7. Clipboard

Double-tap → URL. Trivial:

val cm = requireContext().getSystemService(Context.CLIPBOARD_SERVICE) as ClipboardManager
cm.setPrimaryClip(ClipData.newPlainText("xkcd-url", comic.pageUrl))

Triple-tap → image. Subtler. ClipData.newUri() needs a content:// URI from a FileProvider, but plugin-side <provider> manifest entries don't get registered with the OS at runtime — plugins are loaded via DexClassLoader rather than installed as apps. So we route through the host IDE's existing FileProvider authority:

val ctx = requireContext()
viewLifecycleOwner.lifecycleScope.launch {
    val target = withContext(Dispatchers.IO) {        // up to 5 MB — keep off Main
        File(ctx.filesDir, "xkcd_share/last.png").also { source.copyTo(it, true) }
    }
    val authority = "${'$'}{ctx.packageName}.providers.fileprovider"
    val uri = FileProvider.getUriForFile(ctx, authority, target)
    cm.setPrimaryClip(ClipData.newUri(ctx.contentResolver, "xkcd-image", uri))
}

The host's file_provider_paths.xml exposes filesDir via <files-path name="files" path="."/>, which is why this works without any host-side change.

8. Three-tier tooltip

The tab itself surfaces this walkthrough through three tiers:

  • Tier 1 — summary on the tooltip chip.
  • Tier 2 — detail HTML on "See More".
  • Tier 3 — a button that opens this page.
override fun getTooltipEntries(): List<PluginTooltipEntry> = listOf(
    PluginTooltipEntry(
        tag = "xkcd.tab",
        summary = "Random xkcd comic. Tap to roll a new one.",
        detail = "<p>Tap → new · 2-tap → URL · 3-tap → image</p>",
        buttons = listOf(
            PluginTooltipButton(description = "Code walkthrough", uri = "index.html")
        )
    )
)
override fun getTier3DocsAssetPath(): String? = "docs"
