# 0001. Prefer Room for persistence; raw SQLite only for justified exceptions

- **Status:** Accepted
- **Date:** 2026-06-18
- **Deciders:** Code On The Go team

## Context

Code On The Go stores local state — recent projects, tooltips, in-app help, symbol indexes, and assorted IDE settings — on Android devices, entirely offline. Persistence had crept in unevenly: some features use Room, others open `SQLiteDatabase` directly, with no stated default. That inconsistency makes schema changes error-prone and reviews harder.

Room, Android's Jetpack persistence library, manages the schema, generates the row↔object mapping, verifies queries at compile time, and gives us typed DAOs and structured migrations. Those guarantees are worth more than the footprint they cost for the ordinary read/write relational data an IDE keeps. Hand-written SQLite is easy to get subtly wrong (unparameterized queries, ad-hoc migrations, manual cursor mapping) and every such site is bespoke.

## Decision

**Room is the default for persistence.** New relational/queryable local state uses Room — an `@Entity`, a DAO, and a `RoomDatabase` with explicit migrations — provided through Koin. Prefer typed models over hand-rolled cursor mapping.

**Raw SQLite** (`SQLiteDatabase` / `SupportSQLiteOpenHelper`) is reserved for cases where Room's schema ownership and codegen add no value or get in the way. Reach for it only when one of the conditions below holds; otherwise use Room. Non-relational settings still belong in the **filesystem / preferences (DataStore)**, not a database.

### When raw SQLite is justified (the exceptions)

Use raw SQLite only when one of these applies, and say which in the code/PR:

1. **The database is prebuilt and opened read-only.** The schema is produced and owned elsewhere (shipped as an asset or generated by another component), and CoGo only reads it. Room's migration/verification machinery adds nothing when we never create or evolve the schema at runtime.
2. **Performance- and allocation-critical, high-volume data.** We need granular control over the schema, indices, and query shape to minimize object allocations and memory pressure — Room's generated mapping is overhead on the hot path.
3. **A schema shared across a process or component boundary**, where another component (or an external tool) defines the tables and CoGo reads/serves them.

If none of these hold, use Room. "It's a small table" or "I already know SQL" are **not** exceptions.

### Current exceptions in the codebase

| Feature | Location | Why raw SQLite |
|---|---|---|
| Symbol indexing | `lsp/indexing/.../SQLiteIndex.kt` | Performance/allocation-critical; granular schema & query control (condition 2). |
| In-app tooltips | `idetooltips/.../ToolTipManager.kt` | Prebuilt DB opened read-only (condition 1). |
| In-app / plugin help | `plugin-manager/.../documentation/PluginDocumentationManager.kt` | Prebuilt help-content DB (condition 1). |
| Local web server | `app/.../localWebServer/WebServer.kt` | Reads databases (incl. project data) it doesn't own, read-only (conditions 1 & 3). |

The **Recent Projects** feature (`app/.../roomData/recentproject/`, `RecentProjectRoomDatabase`, `@Database version = 4`) is **not** an exception — it uses Room and is the reference example of the default. Extend it (and add new persistence) the same way.

> Note: `idetooltips` still declares Room Gradle deps it doesn't use — remove them (its store is raw SQLite by exception 1). And the `androidx.room:*` strings in `editor`'s `GroovyAutoComplete` are autocomplete suggestions for the *user's* code, not CoGo persistence.

## Consequences

**Positive**
- One default, applied consistently; compile-time query verification and structured migrations on the common path.
- Less bespoke, hand-mapped SQL to review for correctness and injection.
- Exceptions are explicit and justified against stated conditions, not accidental.

**Negative / costs**
- Room's runtime and annotation processing (`kapt`/KSP) add footprint and build time.
- Contributors must justify raw-SQLite use rather than reach for it by habit.

**Follow-ups**
- Remove the unused Room dependencies from `idetooltips`.
- Provide shared helper utilities for the raw-SQLite exceptions so they stay consistent and safe (parameterized queries — see SECURITY.md).

## Alternatives considered

- **Raw SQLite everywhere** (the prior de-facto stance) — rejected: pushes hand-written SQL, migrations, and cursor mapping onto every feature, with no compile-time safety; inconsistent and error-prone.
- **DataStore / SharedPreferences only** — insufficient for relational/queryable data (e.g. recent projects).
- **A third-party ORM** — more dependencies and surface for little gain over Room, which ships with the platform.
