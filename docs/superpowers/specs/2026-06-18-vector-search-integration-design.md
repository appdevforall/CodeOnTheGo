# Vector Search Integration Design

**Date:** 2026-06-18
**Status:** Approved
**Author:** Claude Code

## Overview

This document describes the integration of semantic code search (vector search) from the `convert-plugin-to-library` branch into the `stage` branch. The feature enables developers to search code by meaning rather than exact keywords, using embeddings generated by the local LLM.

## Goals

1. Replace the demo `VectorSearchTestCommand` with production-ready vector search
2. Copy the complete `vector-search` module from `convert-plugin-to-library` to `stage`
3. Integrate vector search into the existing SearchResultFragment UI
4. Implement on-demand indexing (index on first search, cache results)
5. Keep test/debug command for development purposes

## Non-Goals

- Background indexing on project open (future enhancement)
- Real-time re-indexing on file changes (future enhancement)
- Migration of smart-completion or speech-to-text features (separate tasks)

### 1. vector-search Module

A standalone library module containing all vector search logic.

**Files to copy from convert-plugin-to-library:**

  - `CodeEmbeddingDescriptor.kt` - SQLite schema for CodeEmbedding
  - `VectorMath.kt` - Cosine similarity utilities
  - `VectorSearchService.kt` - Main search API
  - `EmbeddingIndexingService.kt` - Indexing service (copy but may not use for MVP)
- `vector-search/src/test/` - All unit tests

**Dependencies:**
```kotlin
dependencies {
```

### 2. VectorSearchService

**Location:** `vector-search/src/main/kotlin/.../VectorSearchService.kt`

**Primary API:**
```kotlin
suspend fun search(
    query: String,
    limit: Int = 20,
    threshold: Float = 0.0f
): List<CodeEmbedding>
```

**Responsibilities:**
1. Generate embedding for search query using ILlamaController
2. Query SQLiteIndex for all stored CodeEmbeddings
3. Calculate cosine similarity between query and each code chunk
4. Filter by similarity threshold
5. Sort by similarity (descending)
6. Return top K results

**Implementation notes:**
- Thread-safe: runs on Dispatchers.Default
- Handles empty query with IllegalArgumentException
- Returns empty list if no embeddings exist in index
- Logs search metrics (query dimension, result count)

### 3. CodeChunker

**Location:** `vector-search/src/main/kotlin/.../CodeChunker.kt`

**Primary API:**
```kotlin
fun chunkFile(file: File): List<CodeChunk>

data class CodeChunk(
    val text: String,
    val startLine: Int,
    val endLine: Int
)
```

**Chunking strategy:**
- Target chunk size: 200-500 characters
- Chunks align with logical boundaries (functions, classes, XML elements)
- Preserves context (includes some surrounding code)
- Language-aware parsing for Kotlin, Java, XML

### 4. CodeEmbedding

**Location:** `vector-search/src/main/kotlin/.../CodeEmbedding.kt`

**Data model:**
```kotlin
data class CodeEmbedding(
    override val key: String,           // "{filePath}:{chunkIndex}"
    override val sourceId: String,      // filePath (for bulk deletion)
    val filePath: String,
    val chunkText: String,
    val language: String,               // "kotlin", "java", "xml"
    val chunkIndex: Int,                // 0-based chunk number
    val startLine: Int,                 // 1-based line number
    val endLine: Int,                   // 1-based line number (inclusive)
    val embedding: FloatArray           // 384-dimensional vector
) : Indexable
```

**Storage:**
- Persisted in SQLiteIndex at `{cacheDir}/embeddings.db`
- Schema defined in CodeEmbeddingDescriptor
- Indexed by key and sourceId for efficient queries

### 5. VectorSearchCommand (Production)

**Location:** `app/src/main/java/.../api/commands/VectorSearchCommand.kt`

**Purpose:** Replace VectorSearchTestCommand with production implementation

**API:**
```kotlin
class VectorSearchCommand(
    private val query: String,
    private val llmEngine: LlmInferenceEngine,
    private val limit: Int = 20
) : Command<List<CodeEmbedding>>

override fun execute(): ToolResult
```

**Implementation flow:**
1. Validate model is loaded
2. Get or create VectorSearchService instance
3. Check if project is indexed (SQLiteIndex has entries)
4. If not indexed:
   - Get project directory from IProjectManager
   - Collect all .kt/.java/.xml files
   - For each file:
     - Chunk using CodeChunker
     - Generate embeddings using llmEngine
     - Store in SQLiteIndex
5. Perform search using VectorSearchService
6. Return results as ToolResult

**Error handling:**
- Model not loaded → return failure with message
- Project dir not found → return failure
- Embedding generation fails → skip file, log warning
- Search fails → return failure with exception details

### 6. EditorViewModel Integration

**Location:** `app/src/main/java/.../viewmodel/EditorViewModel.kt`

**New method:**
```kotlin
suspend fun performSemanticSearch(query: String)
```

**Implementation:**
```kotlin
suspend fun performSemanticSearch(query: String) {
    viewModelScope.launch {
        try {
            // Get LLM engine
            val engine = LlmInferenceEngineProvider.instance

            // Ensure model is loaded
            if (!engine.isModelLoaded) {
                // Show error: "No model loaded"
                return@launch
            }

            // Execute vector search command
            val command = VectorSearchCommand(query, engine, limit = 20)
            val result = withContext(Dispatchers.IO) {
                command.execute()
            }

            // Convert CodeEmbedding results to SearchResult
            if (result.success && result.data != null) {
                val searchResults = convertToSearchResults(result.data)
                _vectorSearchResults.value = searchResults
            } else {
                // Handle failure
                _vectorSearchResults.value = emptyMap()
            }
        } catch (e: Exception) {
            Log.e("EditorViewModel", "Vector search failed", e)
            _vectorSearchResults.value = emptyMap()
        }
    }
}

private fun convertToSearchResults(
    embeddings: List<CodeEmbedding>
): Map<File, List<SearchResult>> {
    return embeddings
        .groupBy { File(it.filePath) }
        .mapValues { (file, chunks) ->
            chunks.map { chunk ->
                SearchResult(
                    file = file,
                    lineNumber = chunk.startLine,
                    columnNumber = 0,
                    length = chunk.chunkText.length,
                    match = chunk.chunkText.take(100) // Preview text
                )
            }
        }
}
```

**Integration point:**
- Called from search UI when user performs semantic search
- Updates `vectorSearchResults` StateFlow
- SearchResultFragment observes and displays results

## On-Demand Indexing Strategy

### First Search Flow

```
User enters "authentication" in search
    ↓
UI calls EditorViewModel.performSemanticSearch("authentication")
    ↓
VectorSearchCommand checks if project indexed
    ↓
SQLiteIndex.query(ALL) returns empty
    ↓
Show "Indexing project for semantic search..."
    ↓
Scan project directory for .kt/.java/.xml files
For each file:
    ├─ CodeChunker.chunkFile() → chunks
    ├─ For each chunk:
    │   ├─ llmEngine.generateEmbeddings(chunk.text) → embedding
    │   └─ SQLiteIndex.index(CodeEmbedding(...))
    └─ Log progress
    ↓
Indexing complete
    ↓
VectorSearchService.search("authentication")
    ↓
Display results in SearchResultFragment vector section
```

### Subsequent Searches

```
User enters "database" in search
    ↓
EditorViewModel.performSemanticSearch("database")
    ↓
VectorSearchCommand checks if project indexed
    ↓
SQLiteIndex.query(ALL) returns existing embeddings
    ↓
VectorSearchService.search("database") (immediate)
    ↓
Display results (no indexing delay)
```

### Persistence

- Embeddings stored in `{cacheDir}/embeddings.db`
- Survives app restarts
- Cleared when:
  - User clears app cache
  - Manual re-index triggered (future feature)

## UI Integration

### Search Trigger Mechanism

**How search is initiated:**
When user enters a search query and presses search (existing search UI flow), the system will:
1. Run keyword search (existing behavior) → updates `searchResults` StateFlow
2. Run vector search (new behavior) → updates `vectorSearchResults` StateFlow
3. Both results display simultaneously in SearchResultFragment

**Implementation approach:**
- Modify the existing search trigger point to call BOTH search methods
- Keyword search runs synchronously (fast)
- Vector search runs asynchronously (may take time on first search)
- Results appear in their respective sections as they complete

### SearchResultFragment (Existing)

Already has two sections:
1. **Keyword Results** - Existing grep-based search
2. **Vector Results** - Placeholder, now populated with real data

**Changes required:**
- None! UI already observes `EditorViewModel.vectorSearchResults`
- When `performSemanticSearch()` is called, results automatically populate the vector section
- Both sections work independently - keyword results appear immediately, vector results appear when ready

**Keep existing "Test Vector Search" option:**
- Menu item ID: `R.id.menu_test_vector_search`
- Triggers VectorSearchTestCommand (debugging tool)
- Logs detailed results to logcat
- Does NOT update UI (stays as debug tool)

**Why keep it:**
- Useful for testing embedding quality

## Data Flow

### Search Execution Flow

```
User Input: "authentication"
    ↓
SearchResultFragment
    ↓
EditorViewModel.performSemanticSearch("authentication")
    ↓
VectorSearchCommand.execute()
    ├─ Check model loaded (LlmInferenceEngine)
    ├─ Check project indexed (SQLiteIndex)
    ├─ If not indexed:
    │   ├─ Scan project files
    │   ├─ Chunk files (CodeChunker)
    │   ├─ Generate embeddings (LlmInferenceEngine)
    │   └─ Store in SQLiteIndex
    └─ VectorSearchService.search("authentication")
        ├─ Generate query embedding
        ├─ Load all embeddings from SQLiteIndex
        ├─ Compute cosine similarity for each
        ├─ Filter by threshold (0.0)
        ├─ Sort by similarity (desc)
        └─ Return top 20 results
    ↓
Convert CodeEmbedding → SearchResult
    ↓
Update EditorViewModel.vectorSearchResults
    ↓
SearchResultFragment observes and displays in vector section
```

### Indexing Data Flow

```
File: MainActivity.kt
    ↓
CodeChunker.chunkFile(file)
    ├─ Parse file (language-aware)
    ├─ Split into semantic chunks
    └─ Return List<CodeChunk>
        Example chunks:
        - Chunk 0: "class MainActivity : AppCompatActivity() { ... }" (lines 10-30)
        - Chunk 1: "override fun onCreate(...) { ... }" (lines 32-45)
        - Chunk 2: "private fun setupUI() { ... }" (lines 47-60)
    ↓
For each chunk:
    ↓
    LlmInferenceEngine.generateEmbeddings(chunk.text)
        ├─ Tokenize text
        ├─ Run through encoder model
        └─ Return FloatArray[384]
    ↓
    Create CodeEmbedding(
        key = "MainActivity.kt:0",
        sourceId = "MainActivity.kt",
        filePath = "/path/to/MainActivity.kt",
        chunkText = chunk.text,
        language = "kotlin",
        chunkIndex = 0,
        startLine = chunk.startLine,
        endLine = chunk.endLine,
        embedding = FloatArray[384]
    )
    ↓
    SQLiteIndex.index(codeEmbedding)
        └─ INSERT INTO embeddings (...) VALUES (...)
```

## Error Handling

### Model Not Loaded
- **When:** User searches before loading a model
- **Action:** Show toast "No model loaded. Please select a model in AI Settings."
- **Recovery:** User goes to AI Settings, loads model, tries again

### Indexing Failure
- **When:** File can't be read or embedding fails
- **Action:** Log warning, skip file, continue indexing
- **Result:** Partial index (some files indexed, others skipped)

### Search Failure
- **When:** Exception during search execution
- **Action:** Log error, return empty results
- **UI:** Show "Search failed" message

### Empty Results
- **When:** No semantic matches found
- **Action:** Show "No results found" in vector section
- **Note:** Normal behavior, not an error

## Performance Considerations

### Indexing Performance
- **First search:** 5-30 seconds depending on project size
- **Bottleneck:** Embedding generation (model inference)
- **Optimization:**
  - Limit to 100 files for MVP
  - Skip build/ and .gradle/ directories
  - Process files in parallel (future enhancement)

### Search Performance
- **Typical search:** 100-500ms for 1000 embeddings
- **Bottleneck:** Cosine similarity computation (O(n) where n = # embeddings)
- **Optimization:**
  - Use FloatArray for efficient computation
  - Run on Dispatchers.Default (background thread)
  - Consider FAISS/Annoy for large projects (future enhancement)

### Memory Usage
- **Embedding storage:** ~1.5KB per chunk (384 floats × 4 bytes)
- **Typical project:** 1000 chunks = ~1.5MB in memory
- **SQLite storage:** Similar size on disk
- **Acceptable:** Within mobile memory constraints
